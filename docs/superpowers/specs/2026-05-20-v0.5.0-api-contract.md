# Burrow v0.5.0 Backend API & Protocol Contract

> Source of truth for the UI, backend, and integration agents — same role the
> `2026-05-19-v0.4.0-api-contract.md` doc played for v0.4.0. Every shape below
> is either the **target** the v0.5.0 backend plan implements, or is copied
> from an already-shipped v0.4.0 handler. Where this doc and shipped code
> disagree, **the code wins** — open a fix to this doc (the integration
> plan's Task 1, mirroring the v0.4.0 rule).
>
> **Scope (honors `docs/BACKLOG_0.5.0.md`):** v0.5.0 is the **bridge release**
> between v0.4.0 (AI-gateway depth + company-scale) and v1.0.0 (the "boring"
> stability promise). It (1) **matures the AI gateway** — semantic prompt
> cache, upstream-credential injection, multi-provider passthrough; (2)
> **finishes the tunnel polish** carried over from ROADMAP Phase 4 — custom
> domains per tunnel (CNAME, operator-supplied TLS) and per-tunnel connection
> logs; (3) **lands two v1.0 foundations early as alpha / opt-in** —
> Postgres optional backend and retention & compliance knobs; (4) **polishes
> the operator surface** — embedded OpenAPI viewer and expanded webhook event
> vocabulary with operator-customisable payload templates.
>
> Out-of-scope, deferred to v0.6 / v1.0 / v1.1 (do **not** build against
> these): OIDC single-IdP (v1.0); SAML / organisations / multi-tenant
> workspaces (v1.1+); multi-relay clustering (v1.1+); native installers /
> security audit / docs site / systemd templates / one-click VPS deploys /
> reproducible builds (v1.0 launch package); UDP / P2P / IPv6 e2e / HTTP/3
> / k8s operator / Docker plugin / Caddy/Traefik modules / desktop GUI /
> mobile / plugin system (v1.1+); persistent request-inspector storage
> (v0.6+ — intentional retention surface).
>
> **Hard constraints (carried through every part):**
> - Single binary, **no phone-home** (semantic cache embeddings, upstream
>   credentials, connection logs, OpenAPI viewer assets, and retention rules
>   all live on the operator's box).
> - **Apache-2.0 only**; **no new heavy deps** in the default build (see Q-deps
>   open questions; opt-in build tags acceptable, same pattern as the v0.4.0
>   `geo` tag).
> - The SSE/token-stream no-buffering rule
>   (`ReverseProxy.FlushInterval=-1`, v0.3.0) is preserved by every new
>   middleware step. Semantic cache and credential injection MUST NOT buffer
>   the visitor's stream.
> - **APPEND-ONLY** commits on shared `main` — the v0.3.0 `fbebf23` UI-loss
>   and the v0.4.0 `dfc2955` cross-tree-slip lessons. Never amend / reset /
>   rebase across the previous commit; review fixes ship as a follow-up.
> - Upstream-credential storage is `*_FILE`/env only (no DB); same shape as
>   `BURROW_SMTP_PASSWORD`. Encrypted-at-rest is deferred → v1.0+ (Q2).
> - Postgres backend ships as **alpha** behind `experimental:` in config;
>   SQLite remains the default. v1.0 will drop the flag (Q3).
> - Webhook payload templates are sandboxed Go `text/template`s with a
>   **closed function set** — no `os`, `exec`, file I/O, network, or env
>   access (Q9 details).
>
> **Audit/PII pin:** the v0.4.0 audit hash-chain schema is **unchanged** in
> v0.5.0. New event types (Part L) extend the closed `action` vocabulary.

## Conventions (unchanged from v0.4.0 — restated for diffs only)

- **Base path:** `/api/v1`. Health probes (`/healthz`, `/readyz`) unprefixed.
- **Dashboard auth:** `burrow_session` cookie (HttpOnly) OR WebAuthn (v0.4.0).
- **Automation auth:** `Authorization: Bearer bua_<token>` (v0.4.0 Part M).
- **CSRF:** unchanged.
- **Error body:** always `{"error":"<message>"}`. v0.5.0 may add `{"error",
  "reason"}` for IP/geo-style structured denials; the `error` field stays.
- **Migrations:** v0.5.0 adds migrations `0011` through `0017` — one per
  feature family. The full list is in Part M.
- **Build tags:** v0.4.0 introduced `geo`; v0.5.0 adds `semantic_cache` and
  `postgres`. Each tag pulls in one optional module — see Part 0.

## Part A — Semantic prompt cache (extends v0.4.0 Part B.3)

v0.4.0 ships exact-match. v0.5.0 layers a semantic tier **on top**, gated
by the `semantic_cache` build tag and by a per-service config flag.

### A.1 Pipeline (PINNED)

When a request reaches step 6 of the v0.4.0 AI middleware chain
(`cache.lookup()`):

1. **Exact lookup** runs first (v0.4.0 behavior). On exact hit → `HIT`,
   short-circuit. Otherwise:
2. If `semantic.enabled` for the service AND request body is JSON with a
   non-empty `prompt`/`messages[].content` field, compute a query embedding.
   Else fall through to the v0.4.0 MISS path.
3. **ANN search** over the per-service vector index (one collection per
   service in chromem-go terms; serialised on disk under `<DataDir>/cache/
   semantic/<service_id>/`). The index stores tuples
   `(exact_cache_key, prompt_embedding, prompt_fingerprint)` — bodies are
   **not** duplicated; the semantic index points back to the exact-cache row.
4. Top-1 result with cosine similarity ≥ `min_similarity` (**default 0.85**,
   range 0.0–1.0 inclusive; PINNED — see Q4) is a candidate match.
5. **Fallback policy** decides:
   - `treat_as_miss` (default): semantic candidate is dropped; proxy as a
     normal MISS, then optionally promote the response into the exact cache
     under both the canonical key *and* the semantic index.
   - `return_cached_marked`: the candidate's exact-cache entry is served
     **with a `Burrow-Cache: similar` header** (instead of `HIT`) plus
     `Burrow-Cache-Similarity: 0.<87>` so the client knows.
6. On any embedding-model error (timeout, OOM, model not loaded), the
   pipeline silently degrades to the v0.4.0 MISS path; the failure is
   counted into `burrow_ai_semantic_errors_total` (Part K).

### A.2 Embedding source

Operator-pluggable, **two modes only in v0.5.0**:

- **`local`** — a sentence-transformer model loaded by the operator into
  Ollama (or any OpenAI-compatible local endpoint) and addressed by URL +
  model name. Burrow calls it via the same `internal/proxy` egress dialer,
  with a 250 ms hard timeout. The embedding result vector is stored
  verbatim (no normalisation beyond the model's own output, which
  sentence-transformer models already normalise).
- **`none`** — the semantic tier is disabled regardless of the build tag
  or per-service config (operator hard-off).

**No cloud-embedding mode in v0.5.0.** Sending visitor prompts to a third
party for embedding would violate the "no phone-home" promise. v0.6+ may
add an opt-in cloud-embedding mode with explicit operator consent UI.

### A.3 Config (extends v0.4.0 `ServiceAIConfig.cache`)

```json
"cache": {
  // v0.4.0 fields (unchanged):
  "enabled": false, "applies_per": "per_endpoint",
  "ttl_seconds": 600, "max_entries": 1000, "max_per_entry_kb": 64,

  // v0.5.0 additions:
  "semantic": {
    "enabled":            false,                 // off by default — same as v0.4.0
    "min_similarity":     0.85,                  // 0.0..1.0; PINNED default
    "embedding_mode":     "local",               // "local" | "none"
    "embedding_url":      "http://localhost:11434/v1/embeddings",
    "embedding_model":    "nomic-embed-text",
    "fallback_policy":    "treat_as_miss",       // "treat_as_miss" | "return_cached_marked"
    "promote_on_miss":    true,                  // index new entries into the vector store
    "max_index_entries":  10000                  // per-service cap; LRU eviction same as exact
  }
}
```

`PUT /api/v1/services/{id}/ai-config` validates `semantic.min_similarity ∈
[0.0, 1.0]`; out-of-range → `400 {"error":"semantic.min_similarity out of
range"}`. `embedding_mode` must be in the closed set above; unknown values
→ `400 {"error":"unknown embedding_mode \"<x>\""}`.

### A.4 JSON API additions

`GET /api/v1/cache/settings` extends with a top-level `semantic` block
matching the shape above (global defaults shape only; per-service settings
ride on `ServiceAIConfig`). `GET /api/v1/cache/stats` extends:

```json
{
  // v0.4.0 fields:
  "entries": 47, "on_disk_bytes": 3145728, "hit_rate_24h": 0.31,
  // v0.5.0 additions:
  "semantic_entries":     12,
  "semantic_disk_bytes":  524288,
  "semantic_hit_rate_24h": 0.04,
  "semantic_similar_returned_24h": 7,   // returns under return_cached_marked
  "semantic_promotions_24h": 3
}
```

`DELETE /api/v1/cache/entries` clears **both tiers** (exact + semantic). A
new `DELETE /api/v1/cache/semantic/entries` clears semantic only (admin or
`ai:configure:any`).

### A.5 Wire headers (visitor-facing)

| Header | When | Value |
|---|---|---|
| `Burrow-Cache` | Always (v0.4.0) | `HIT` \| `MISS` \| `SKIP` \| `similar` (NEW) |
| `Burrow-Cache-Similarity` | When `Burrow-Cache: similar` only (NEW) | `0.85`–`1.00` (the matched score) |
| `Burrow-Cache-Age` | v0.4.0 | unchanged |

### A.6 Migration (Part M migration 0011)

```sql
-- 0011_v0.5.0_semantic_cache.sql
CREATE TABLE semantic_index (
  service_id        TEXT NOT NULL REFERENCES services(id) ON DELETE CASCADE,
  exact_key_hash    TEXT NOT NULL,             -- joins back to cache_entries
  prompt_fingerprint TEXT NOT NULL,            -- sha256 of canonical prompt text
  embedding_dim     INTEGER NOT NULL,
  embedding_blob    BLOB NOT NULL,             -- float32 little-endian
  created_at        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (service_id, exact_key_hash)
);
CREATE INDEX idx_semantic_service_ts ON semantic_index(service_id, created_at DESC);
```

The chromem-go in-memory index is **rebuilt from this table on startup**;
the table is the durable record. The vector store is **never** the source
of truth (chromem-go is an accelerator, not a database).

## Part B — Upstream-credential injection (NEW)

The operator stores the real OpenAI / Anthropic / etc. key **once**, in
the `*_FILE`/env channel (never in DB). Visitors authenticate with a
Burrow-minted key (the v0.3.0 + v0.4.0 access-mode flow). On the egress
path the proxy strips the visitor's API-key header and injects the
upstream key. One real cloud key behind the relay; per-app Burrow keys
out front.

### B.1 Storage (PINNED — vault scheme decision)

Per spec Q2: **`*_FILE`/env only**, no DB row, no encrypted-at-rest.
Named slots:

```
BURROW_UPSTREAM_KEY_<SLOT>
BURROW_UPSTREAM_KEY_<SLOT>_FILE
```

`<SLOT>` is `[A-Z0-9_]{1,32}` — operator-chosen handle. Examples:
`BURROW_UPSTREAM_KEY_OPENAI`, `BURROW_UPSTREAM_KEY_ANTHROPIC_TEAM_A`.
The `_FILE` variant reads from the path with the same trim semantics as
`BURROW_SMTP_PASSWORD_FILE` (existing v0.2.0 `applyFileSecrets`).

At startup, Burrow scans `os.Environ()` for matching prefixes and
populates an in-process map `slot -> plaintext`. The plaintext is held
ONLY in process memory and is **never** logged, **never** returned by
any JSON API, **never** written to DB or to the audit payload. The
audit log records only the slot **handle** (e.g. `slot:OPENAI`).

### B.2 Per-service binding (JSON API)

A new table binds a service to a slot:

```sql
-- 0012_v0.5.0_upstream_credentials.sql
CREATE TABLE service_upstream_credentials (
  service_id     TEXT PRIMARY KEY REFERENCES services(id) ON DELETE CASCADE,
  slot           TEXT NOT NULL,                -- e.g. "OPENAI" (no value, just handle)
  header_name    TEXT NOT NULL DEFAULT 'Authorization',
  header_format  TEXT NOT NULL DEFAULT 'Bearer {key}',
  created_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

| Method | Path | Request | Success |
|---|---|---|---|
| GET | `/api/v1/upstream-credentials/slots` | — | `200 {slots: ["OPENAI", "ANTHROPIC_TEAM_A"]}` (handles only, no values) |
| GET | `/api/v1/services/{id}/upstream-credential` | — | `200 {slot, header_name, header_format, slot_present: true}` or `200 {slot_present: false}` if unbound |
| PUT | `/api/v1/services/{id}/upstream-credential` | `{slot, header_name?, header_format?}` | `204` — `400 unknown slot` if the env handle isn't present, `400 invalid header_format` if `{key}` is missing |
| DELETE | `/api/v1/services/{id}/upstream-credential` | — | `204` (clears the binding; subsequent egress requests will fail if the upstream requires a key) |

Permission: `ai:configure:own|any` (existing v0.4.0 permission).

### B.3 Egress injection (data plane, behavioural)

After the v0.4.0 access-mode check passes and before the
`ReverseProxy.Director` rewrites the URL, a new middleware step
`credinject()` runs (between `route()` and `proxy()` in the v0.4.0 chain
ordering):

1. Look up the resolved service's `upstream_credential` row.
2. If unbound or no env slot matches → pass-through unchanged (no error;
   the upstream may still accept the request, e.g. local Ollama).
3. **Strip** the visitor's incoming `header_name` value (default
   `Authorization`) **completely** from the upstream request — never
   forward a Burrow-minted key.
4. **Inject** `header_name: <header_format with {key} → slot.value>`.
   Default format `Bearer {key}`. Other formats: `Bearer {key}` (OpenAI),
   `{key}` (Anthropic uses bare `x-api-key:`), so the format **template
   variable** is `{key}` exactly. `{key}` not present in format →
   reject the PUT with `400`.

The visitor-facing API key (Burrow-minted) is validated by the v0.3.0
access-mode check BEFORE this step ever runs — so credential injection
only happens on already-authorised requests.

**Idempotency-Key behaviour:** the credential injection does NOT modify
the body, so the v0.4.0 idempotency-aware retry rule is preserved.

### B.4 Audit + metrics

- Audit event `service.upstream_credential.{bind,unbind}` (action
  vocabulary in Part L). Payload: `{slot}` only.
- Metric `burrow_ai_credential_injections_total{service, slot}` (Part K).
- Metric `burrow_ai_credential_misses_total{service}` — incremented when
  the binding points at a slot whose env var is absent at request time.

### B.5 Honesty disclosure (UI surfacing requirement)

The dashboard (UI plan Task 5) MUST display the slot **handle** never the
value, and must explicitly note "Stored in env var, not in the database"
on the binding page. This is the operator's audit trail.

## Part C — Multi-provider passthrough (extends v0.4.0 Part C)

v0.4.0 ships routing strategies (`single`/`failover`/`weighted`/
`header_based`/`sticky`) and the model registry (`model_aliases`). All
backends are assumed to live behind one upstream provider.

v0.5.0 extends the registry + the strategies to cross provider boundaries.

### C.1 Model registry extension

The `model_aliases` table (v0.4.0 migration 0009) gains two columns
(migration 0013):

```sql
-- 0013_v0.5.0_multi_provider.sql
ALTER TABLE model_aliases ADD COLUMN provider TEXT NOT NULL DEFAULT '';
ALTER TABLE model_aliases ADD COLUMN priority INTEGER NOT NULL DEFAULT 100;
-- 0 = highest priority; ties broken by alias id
CREATE INDEX idx_model_aliases_alias_priority
  ON model_aliases(alias, priority ASC);
```

Permitted `provider` values (closed set, v0.5.0): `ollama` | `vllm` |
`openai-compat` | `openai` | `anthropic` | `other`. The `provider` field
must match the **target service's** detected upstream family — Burrow does
not transcode model names cross-provider (that's a LiteLLM concern).

Multiple aliases can share the same `alias` string with different
`provider` + `priority`; the **router** picks one based on the strategy.

### C.2 New strategy: `multi_provider`

Per-service `routing.strategy` gains a new value `multi_provider`.
Semantics:

1. Resolve the request's `model` against all aliases sharing that name.
2. Order by `(priority ASC, alias_id ASC)`.
3. **Walk in order**, skipping any alias whose backing service's circuit
   breaker is open (v0.4.0 Part C.3) or whose `provider` doesn't match the
   request's translation rule (Q5).
4. **Translation policy** (per-service `routing.translate_to`, v0.4.0):
   - `none` (default): each alias's `provider` must match the visitor's
     wire shape exactly. A `chat/completions` request only routes to
     `openai` / `openai-compat` / `ollama` / `vllm` aliases. A
     `/v1/messages` request only routes to `anthropic`.
   - `openai` or `anthropic`: the v0.4.0 Anthropic adapter
     (Part Q) runs in both directions; cross-provider aliases are
     permitted. Lossy fields (tool schemas, image inputs) are flagged in
     the inspector entry as `adapter.lossy=true`.
5. **Mid-stream failover** (PINNED — Q6): forbidden across provider
   boundaries. Cross-provider failover may only happen when (a) the
   upstream returned a connection error before any bytes streamed, or (b)
   the request carries `Idempotency-Key` AND no body bytes have been
   forwarded. **Same rule as v0.4.0 Part C.2 — restated here so cross-
   provider doesn't accidentally widen it.** A POST that has streamed
   even one byte to the visitor MUST NOT retry on a fresh provider; the
   visitor gets the partial stream plus the v0.4.0 trailer event flag.

### C.3 JSON API additions

`POST /api/v1/models/aliases` request body extends:

```json
{
  "alias": "fast",
  "concrete_model": "llama3.1:8b",
  "service_id": "svc_ai001",
  "provider": "ollama",           // v0.5.0 — required
  "priority": 100                  // v0.5.0 — optional, default 100
}
```

`PUT /api/v1/models/aliases/{alias}` accepts `{provider?, priority?}` to
update an existing alias.

`GET /api/v1/models/aliases` returns the v0.5.0 shape with `provider` +
`priority`; old clients that ignore the new fields keep working.

### C.4 Wire metric: provider in failover events

`burrow_ai_failover_events_total{service,from,to}` is extended with a
`cross_provider="true|false"` label (PINNED — operators must be able to
alert on cross-provider failover specifically).

## Part D — Custom domains per tunnel (CNAME + operator-supplied TLS)

v0.3.0 ships subdomains under `auth_domain` (`<sub>.tunnels.example.com`).
v0.5.0 adds operator-owned domains via CNAME — `foo.example.com` →
`<sub>.tunnels.example.com`. The operator supplies the TLS cert for
their domain; Burrow does **not** integrate ACME in v0.5.0 (mirror v0.3.0
Q1 — operator-supplied TLS only).

### D.1 Storage (migration 0014)

```sql
-- 0014_v0.5.0_custom_domains.sql
CREATE TABLE service_custom_domains (
  id             TEXT PRIMARY KEY,
  service_id     TEXT NOT NULL REFERENCES services(id) ON DELETE CASCADE,
  hostname       TEXT NOT NULL UNIQUE,               -- e.g. "foo.example.com"
  cert_pem       TEXT NOT NULL,                      -- full chain PEM
  key_pem        TEXT NOT NULL,                      -- private key PEM
  cert_sha256    TEXT NOT NULL,                      -- fingerprint for display
  not_before     DATETIME NOT NULL,
  not_after      DATETIME NOT NULL,
  created_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at     DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_custom_domains_service ON service_custom_domains(service_id);
CREATE INDEX idx_custom_domains_not_after ON service_custom_domains(not_after);
```

**Privacy note:** `key_pem` is the only PII-grade secret stored in the DB
in v0.5.0. The decision to keep it in the DB (vs. an env-only vault) is
operational: operators expect to manage many domains via the UI, and
env-per-domain doesn't scale. Q7 records the trade-off and the v1.0+
encrypted-at-rest plan.

### D.2 JSON API

| Method | Path | Request | Success |
|---|---|---|---|
| GET | `/api/v1/services/{id}/domains` | — | `200 [Domain]` (no `key_pem` in list) |
| POST | `/api/v1/services/{id}/domains` | `{hostname, cert_pem, key_pem}` | `201 Domain` |
| GET | `/api/v1/services/{id}/domains/{did}` | — | `200 Domain` (still no `key_pem`) |
| PUT | `/api/v1/services/{id}/domains/{did}` | `{cert_pem?, key_pem?}` | `204` |
| DELETE | `/api/v1/services/{id}/domains/{did}` | — | `204` |

```
Domain := {
  id, service_id, hostname,
  cert_sha256, not_before, not_after,
  created_at, updated_at,
  status: "active" | "expiring_soon" | "expired"  // computed, not stored
}
```

`status` derived per request: `expired` if `not_after < now`;
`expiring_soon` if `not_after - now < 14 days`; else `active`.

### D.3 Validation (PINNED)

On POST/PUT:
1. **Parse** `cert_pem` — must contain ≥1 `CERTIFICATE` block; first
   block's `Subject Alternative Name` must include `hostname` (or
   `CommonName` if no SAN; legacy support only — emit a warning header
   `Burrow-Warning: cert-no-san`).
2. **Parse** `key_pem` — must be RSA / ECDSA / Ed25519 private key.
3. **Match** — public key in the first cert must match `key_pem`.
4. **Verify** — chain validates against the **system roots** (Burrow does
   NOT do its own root set; if the operator has a private CA, they trust
   it system-wide).
5. **Reject** with `400 {"error":"...", "reason":"san_mismatch|chain_invalid|key_mismatch|expired"}` —
   structured error so the UI can render the right message.

A `409 {"error":"hostname already bound"}` is returned when another
service already claims the hostname.

### D.4 Data-plane wiring

The existing HTTP reverse-proxy ingress (`:8443`) gains a
`tls.Config.GetCertificate` callback that:

1. Looks up the SNI hostname against `service_custom_domains.hostname`.
2. If found: returns the per-domain cert + key.
3. If not found: falls back to the operator-configured wildcard cert
   (existing v0.3.0 behaviour for subdomains under `auth_domain`).
4. If neither: TLS handshake fails with the standard `unrecognized_name`
   alert.

The vhost router (`internal/proxy`) already routes by `Host:` header;
custom-domain rows extend the same lookup table — same access-mode
checks, same AI middleware chain. **No new code path on the hot path.**

### D.5 Audit + metrics

- `service.custom_domain.{add,update,delete}` in the audit vocabulary.
- `burrow_cert_expiry_days{domain="foo.example.com"}` gauge per custom
  domain (extends the v0.4.0 Part O `burrow_cert_expiry_days{cert}`).
- Webhook event `cert.expiring` (existing v0.4.0) fires per domain when
  `not_after - now < 14 days`, rate-limited to once per domain per day.

## Part E — Per-tunnel connection logs

Distinct from the v0.4.0 request inspector (HTTP-level, ring-buffer
only): connection logs are **TCP-level** session records. One row per
control-protocol session AND one row per HTTP-proxy session.

### E.1 Storage (PINNED — raw + rollup, per Q5)

```sql
-- 0015_v0.5.0_connection_logs.sql
CREATE TABLE connection_logs (
  id              TEXT PRIMARY KEY,                       -- ulid
  kind            TEXT NOT NULL,                          -- 'control' | 'http_proxy' | 'tcp_proxy'
  service_id      TEXT REFERENCES services(id) ON DELETE SET NULL,
  tunnel_id       TEXT,                                   -- v0.3.0 transient id; nullable post-disconnect
  user_id         TEXT REFERENCES users(id) ON DELETE SET NULL,
  client_session_id TEXT,                                 -- yamux session id (control); empty for proxy
  source_ip       TEXT NOT NULL,                          -- trusted-proxy-resolved
  user_agent      TEXT,                                   -- HTTP only
  started_at      DATETIME NOT NULL,
  ended_at        DATETIME NOT NULL,                      -- always populated on insert
  duration_ms     INTEGER NOT NULL,
  bytes_in        INTEGER NOT NULL DEFAULT 0,
  bytes_out       INTEGER NOT NULL DEFAULT 0,
  status          TEXT NOT NULL,                          -- 'closed_clean' | 'closed_error' | 'closed_idle' | 'rejected'
  reason          TEXT,                                   -- optional short label, e.g. 'access_denied'
  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_connlog_started ON connection_logs(started_at DESC);
CREATE INDEX idx_connlog_service_started ON connection_logs(service_id, started_at DESC);
CREATE INDEX idx_connlog_kind_started ON connection_logs(kind, started_at DESC);

-- Daily rollup (kept beyond raw retention; default raw=30d, rollup=∞).
CREATE TABLE connection_log_rollups (
  day              DATE NOT NULL,
  service_id       TEXT NOT NULL,
  kind             TEXT NOT NULL,
  sessions         INTEGER NOT NULL DEFAULT 0,
  bytes_in         INTEGER NOT NULL DEFAULT 0,
  bytes_out        INTEGER NOT NULL DEFAULT 0,
  avg_duration_ms  INTEGER NOT NULL DEFAULT 0,
  p95_duration_ms  INTEGER NOT NULL DEFAULT 0,
  created_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (day, service_id, kind)
);
```

**Insertion cadence:** rows are inserted on session **close** only (one
insert per session, never per packet — bounded cardinality). For
long-lived control sessions, a 6-hour rotate-and-roll job emits a row
for "session-so-far" and starts a new id (this prevents the audit chain
from being held open against a multi-week control session).

**Rollup cadence:** a background goroutine, at 00:05 UTC daily, computes
the previous day's rollup row per `(service_id, kind)`, then deletes raw
rows older than `connection_logs.retention_days` (default 30; see Part F).

### E.2 JSON API

| Method | Path | Request | Success |
|---|---|---|---|
| GET | `/api/v1/connection-logs` | `?kind=&service_id=&since=&until=&q=&limit=&before_id=` | `200 [ConnectionLog]` (descending start; cursor-paginated) |
| GET | `/api/v1/connection-logs/rollups` | `?service_id=&kind=&since=&until=` | `200 [Rollup]` |
| GET | `/api/v1/connection-logs/export` | `?kind=&since=&until=&format=ndjson|csv` | streamed download |

```
ConnectionLog := {
  id, kind, service_id, tunnel_id, user_id, client_session_id,
  source_ip, user_agent, started_at, ended_at, duration_ms,
  bytes_in, bytes_out, status, reason
}
Rollup := {
  day, service_id, kind, sessions,
  bytes_in, bytes_out, avg_duration_ms, p95_duration_ms
}
```

Permission: `audit:read` (existing v0.4.0; connection logs are
admin-scoped — operator-level data, not visitor-level).

### E.3 Webhook event

`connection.session_summary` — rate-limited rollup, not per-session
(per BACKLOG_0.5.0). The dispatcher emits one event per
`(service_id, kind)` per hour with the rollup of the closed sessions
in that hour. Payload:

```json
{
  "service_id": "svc_ai001",
  "kind": "http_proxy",
  "window_start": "2026-05-20T12:00:00Z",
```

## Part F — Retention & compliance controls

The v0.2.0 `settings` k/v table grows new keys; v0.5.0 sets **deliberate
defaults** so the relay defaults to operator-safe behaviour.

### F.1 Setting keys (added to existing settings table)

| Key | Default | Range | Effect |
|---|---|---|---|
| `audit.retention_days` | `0` (=∞ forever) | `0..3650` | Rows older than N days are deleted by the daily compaction job; `0` = keep forever (SQLite handles millions fine — honest position) |
| `inspector.retention_count` | `100` | `1..1000` | Ring buffer size per service (already used by v0.4.0); confirmed here for completeness |
| `usage.retention_days` | `90` | `1..3650` | `usage_events` rows older than N days are deleted daily |
| `redaction.retention_days` | `30` | `1..3650` | Audit events with action `redaction.applied` older than N days are deleted (the aggregated form — keeps the chain valid since these are leaf events; rule defended by Q8) |
| `connection_logs.retention_days` | `30` | `1..3650` | Raw `connection_logs` rows older than N days are deleted (rollups stay) |
| `connection_logs.rollups_retention_days` | `0` (=∞) | `0..3650` | Rollup rows older than N days are deleted; `0` keeps forever |
| `webhook_deliveries.retention_days` | `30` | `1..365` | Old delivery records deleted daily (the webhooks themselves are forever) |

**Reading:**

| Method | Path | Success |
|---|---|---|
| GET | `/api/v1/settings/retention` | `200 {audit_retention_days: 0, usage_retention_days: 90, ...}` (read-only view over the keys above) |
| PUT | `/api/v1/settings/retention` | `204` (validates ranges; admin or `settings:manage`) |

The `settings:manage` permission is already in the v0.4.0 vocabulary.

### F.2 Compaction job (PINNED behavioural details)

A single background goroutine, kicked at `:00:30` UTC daily by `time.
NewTicker`. The job:

1. Loads the seven retention settings (one round-trip).
2. For each non-zero setting, computes `cutoff_ts = now() - days*24h`.
3. Runs **one DELETE per table** with `WHERE created_at < ?` (or the
   equivalent timestamp column per table). Logged via `slog` with
   `kind=retention.compact rows_deleted=<n>`.
4. For `audit.retention_days`: SPECIAL HANDLING — deleting audit rows
   would break the hash chain. The job DELETES only **aggregate-eligible**
   action keys (`redaction.applied`, `guardrail.refused` — see Q8); all
   other audit rows are retained regardless of `audit.retention_days`
   (so the chain stays unbroken). The doc warns the operator (`GET
   /api/v1/settings/retention` returns an `audit_retention_note`
   advisory string).

The job is idempotent — running twice in one day deletes zero rows the
second time. On any single-table error the job logs and continues to the
next table.

### F.3 Audit event

`retention.compact` is added to the audit vocabulary (Part L), with
payload `{table: "...", rows_deleted: N}`. Rate-limited: at most one
event per `(table)` per day (the job runs daily).

## Part G — Postgres optional backend (alpha)

v0.5.0 ships the abstraction + the `postgres` build tag + a single
parity-tested migration set. SQLite remains the default. v0.5.0 marks
this **alpha** in the config + release notes.

### G.1 Config

```yaml
# config: ServerConfig additions
database_url: "postgres://user:pass@host:5432/burrow?sslmode=verify-full"  # optional
experimental:
  postgres_backend: true   # MUST be explicitly true to use database_url
```

Both fields default empty/false. When `database_url` is set AND
`experimental.postgres_backend == true`, Burrow uses the Postgres backend;
otherwise it uses the existing SQLite path (`database_path`). Mutually
exclusive: setting both `database_path` and `database_url` is a fatal
config error.

`database_url` MUST come from env (`BURROW_DATABASE_URL` /
`BURROW_DATABASE_URL_FILE`) since it contains a password; the
`*_FILE` variant is recommended.

### G.2 Migration parity (PINNED — Q3)

**Decision: parallel migration set, not portable subset.** Per Q3, v0.5.0
ships:

```
internal/db/migrations/0001_init.sql                 (SQLite, existing)
internal/db/migrations/0001_init.postgres.sql        (NEW)
internal/db/migrations/0002_v0.2.0_foundation.sql                (existing)
internal/db/migrations/0002_v0.2.0_foundation.postgres.sql       (NEW)
...
internal/db/migrations/0011_v0.5.0_semantic_cache.sql            (NEW SQLite)
internal/db/migrations/0011_v0.5.0_semantic_cache.postgres.sql   (NEW Postgres)
... (one per v0.5.0 migration family)
```

The migration runner (`internal/db/migrate.go`) selects the file
extension based on the backend driver:

- SQLite driver → loads `.sql` (existing)
- Postgres driver → loads `.postgres.sql`

A startup self-test (Task 14) runs both ladders against ephemeral DBs and
asserts the resulting `information_schema` is **structurally compatible**
(same tables, columns, indexes; column types may differ between
`DATETIME` and `TIMESTAMP WITH TIME ZONE`). The test fails the build if
a v0.5.0+ migration ships without its Postgres twin.

### G.3 Backend abstraction (Go-internal)

`internal/db/db.go` grows a small interface:

```go
type Backend interface {
    DB() *sql.DB                   // existing field accessor
    Driver() string                // "sqlite" | "postgres"
    Now() string                   // SQL fragment: "CURRENT_TIMESTAMP" or "now()"
    PlaceholderFmt() string        // "?" or "$N"
}
```

Existing typed accessors (`internal/db/users.go`, `services.go`, etc.)
are **rewritten** to consume `Backend` rather than `*sql.DB`. The
rewrite is **mechanical** (search-and-replace `?` → `b.PlaceholderFmt()`)
and goes in **one task** (backend Task 15) — see plan for the
disciplined approach.

### G.4 Feature parity in v0.5.0 (PINNED)

Every v0.4.0 + v0.5.0 feature **must work** on Postgres. No SQLite-only
features. The integration plan's release pass runs the **same** e2e
suite against both backends.

**Not in v0.5.0** (deliberately): multi-relay clustering (the Postgres
backend is single-relay only in v0.5.0, just like SQLite — review Part 7).

### G.5 Honesty disclosure (UI surfacing requirement)

The dashboard's new "Database backend" panel (UI Task 6) MUST display
the driver name, an "alpha" badge, and a link to the migration parity
test results. This is the early-field-testing surface.

### H.3 Payload templates (PINNED — Q9)

A new per-webhook `payload_template` field lets the operator customise
the JSON body. v0.5.0 uses stdlib `text/template` with a **closed
function set** — no `os`, `exec`, file I/O, network, or env access. The
allowed functions are:

| Function | Purpose | Origin |
|---|---|---|
| `printf` | format strings | stdlib |
| `lower`, `upper`, `title` | case mapping | stdlib `strings` |
| `replace`, `trim`, `split` | string ops | stdlib `strings` |
| `now` | current UTC RFC3339 | injected wrapper |
| `default` | `{{default "x" .Y}}` | sprig-style mini-impl, ≈10 LOC, NOT a sprig dep |
| `b64enc`, `b64dec` | base64 | stdlib |
| `json` | `{{json .Payload}}` → JSON-encoded value | injected wrapper |

**Forbidden** by sandbox: nested template includes (`template "x"`),
`call`, `index` with arbitrary keys (only on the well-known fields), any
function not in the table above. Templates are compiled at webhook save
time; an unknown function or syntax error returns `400 {"error":
"template: <Go error>"}`.

Each event has a documented set of fields available; templates that
reference undefined fields render as the empty string (stdlib default).

optional `payload_template` field (Go template source as a string).

response — see v0.4.0).

`POST /api/v1/webhooks/{id}/preview` is new in v0.5.0:

```

→ 200 { "rendered": "{...}", "size_bytes": 184 }
```

### J.2 Implementation choice (PINNED — Q10)

`swagger-ui-dist`:
- ≈3 MB unpacked; embedded into the binary that's currently ≈25 MB

- MIT (compatible), but pulls in dependencies on bundled fonts and
  bundled SVG icons.

```
internal/openapi/viewer/

  viewer.css         # ≈1 KB — same DS variables as the dashboard
  viewer.js          # ≈4 KB — fetches /api/v1/openapi.yaml, renders
  README.md          # development notes
```

Total ≈8 KB embedded. The JS dependency is **zero**: parses YAML with a
tiny regex-based recogniser (≈80 lines — OpenAPI v3 has fully-defined
keys, so a recogniser is feasible). Out-of-scope features (try-it-out,
auth flows) are deferred to v0.6+.

The README acknowledges this is intentionally lightweight: operators
who want a richer experience may fetch `/api/v1/openapi.yaml` and load
it into their preferred local tool.

### J.3 No network calls (PINNED)

The viewer JS MUST NOT make any external HTTP calls. The integration
plan adds a Playwright assertion that the rendered page makes exactly
ONE same-origin fetch (`/api/v1/openapi.yaml`) and nothing else. This
guards the no-phone-home promise as a continuous test.

## Conventions for the integration plan

- Spec ⟷ code wins. Where shipped Go disagrees with this doc, the
  integration plan's Task 1 fixes the doc.
- Default-build CI runs `go test -tags=''`. The two tag-gated test files
  (`semantic_cache_test.go`, `postgres_test.go`) are excluded from the
  default build via `//go:build semantic_cache` / `//go:build postgres`.
- Postgres parity: the integration plan's release pass runs the e2e

  `docker-compose` ephemeral container in CI; locally an operator can
  use `BURROW_DATABASE_URL=postgres://...` directly).
- Webhook templates: the integration plan adds a fuzz harness that
  feeds malformed template sources to the `POST /webhooks` validator and
  asserts the sandbox holds (no `os.Exit`, no goroutine leak, no
  unbounded memory). The fuzz seed corpus lives in
  `internal/webhook/testdata/template_fuzz/`.

## Open Questions (surfaced, not papered over)

- **Q1 — Semantic-cache vector store.** Two candidates considered:
  - **`chromem-go`** (pure Go, Apache-2.0 — verified upstream;
    in-memory + disk persistence; ≈3.5k LOC). Adopted under
    `semantic_cache` build tag (see Part 0 + Part A).
  - **Hand-rolled flat index + brute-force cosine.** Viable up to ~10k
    entries (single-core, ≈1 ms search at 10k×768-dim). Rejected because
    the v0.5.0 default `max_index_entries` is 10k per service and a
    busy 10-service relay would push 100k entries; flat-index would
    cost ~10 ms per request. The chromem-go HNSW index scales to ~1M
    entries with ~1 ms search.
  - **`sqlite-vec`** considered and rejected: requires CGO, fails the
    pure-Go single-binary bar.

  Downgrade if maintainer rejects `chromem-go`: drop semantic cache to
  v0.6; keep the `Semantic` interface in `internal/cache/semantic/`
  with the `NoopSemantic` impl only.

- **Q2 — Upstream-credential vault encryption.** Two options:
  - **`*_FILE`/env only** (Part B.1). Same shape as
    `BURROW_SMTP_PASSWORD`. Pinned for v0.5.0.
  - Encrypted-at-rest in the DB with a relay-wide KEK. Deferred → v1.0+
    when the broader vault concept is ready (managed via a `burrowd
    vault` subcommand mirroring `burrowd audit`).

- **Q3 — Postgres migration parity.** Two options:
  - **Parallel migration files** (one `.sql` + one `.postgres.sql` per
    family). PINNED for v0.5.0 (Part G.2). Explicit, auditable, and the
    v1.0 "boring" promise wants explicitness over cleverness.
  - **Cross-portable subset** (one file per family, restricted DDL).
    Rejected: the existing `0001_init.sql`–`0010_v0.4.0_automation.sql`
    already use SQLite-flavoured DDL (`DATETIME`, `INTEGER` for booleans).
    Retro-fitting them is too much surface for a bridge release.

- **Q4 — Semantic similarity threshold default.** Three points considered:
  `0.80`, `0.85`, `0.90`. Pinned `0.85` — empirical sweet spot per the
  sentence-transformers literature (≥0.85 = "essentially the same
  question"; ≥0.90 = "near-identical paraphrase"; ≥0.80 = "related
  topic"; the gateway use-case prefers high precision over recall).
  Operators override per service.

- **Q5 — Connection-logs cardinality.** Pinned: **raw + rollup, raw
  default-30d / rollup default-∞**. The rollup table holds the long-term
  story (cost: one row per `(day, service_id, kind)` — bounded). The raw
  table is for forensics (cost: one row per session close — operator
  tunable). Operators who want zero-raw can set
  `connection_logs.retention_days: 0` (treated as same-day-only); zero
  raw is acceptable since the rollup still records the aggregate.

- **Q6 — Cross-provider mid-stream idempotency.** Pinned forbidden
  (Part C.2). Cross-provider failover requires `Idempotency-Key` AND
  zero bytes streamed. The wire fields are unchanged from v0.4.0;
  v0.5.0 just widens the *which providers we can pick from* axis, not
  the *when can we retry* axis. The decision is conservative — operators
  who want broader retry semantics can layer LiteLLM in front of Burrow
  for the failover, with Burrow providing trust + privacy + caching.

- **Q7 — Custom-domain TLS shape.** Three options:
  - **Per-domain cert + key in DB** (Part D.1). PINNED for v0.5.0.
    Operational concern: multi-tenant relays may have many domains;
    env-per-domain doesn't scale; the UI wants live updates. The DB row
    holds `key_pem` plaintext — explicit trade-off, documented in the
    UI as "stored in the database; encrypt your database disk if your
    threat model requires it".
  - Multi-SAN wildcard from the operator-supplied `auth_domain`
    wildcard. Rejected: the wildcard fundamentally can't cover
    arbitrary CNAMEs across operator-controlled root domains.
  - ACME / Let's Encrypt. Deferred → v0.3.1 already on the roadmap;
    v0.5.0's slot is consistent with the v0.3.0 Q1 decision.

- **Q8 — Audit retention semantics.** The audit chain is hash-linked
  (each row references `prev_hash`). Deleting middle rows breaks the
  chain. Two reconciled positions:
  - The default `audit.retention_days = 0` (=forever) preserves the
    full chain by default.
  - When operators DO set retention, ONLY rate-limited / aggregated
    leaf rows (`redaction.applied`, `guardrail.refused`,
    `connection.session_summary`, `retention.compact`,
    `ai.cache.promoted`, `ai.upstream_error`) are eligible for
    compaction — these are leaf rows whose deletion doesn't break the
    hash chain by definition (they are never `prev_hash` referenced).
  - The compaction job docstring **lists the eligible action keys**;
    the closed set is hard-coded in `internal/audit/compact.go`.

- **Q9 — Webhook payload-template sandbox.** Three options:
  - **stdlib `text/template`** with a closed function set (Part H.3).
    PINNED for v0.5.0. Familiar to Go operators; no new dep.
  - `html/template` — rejected; HTML auto-escaping mangles JSON.
  - `cel-go` (CEL) — rejected; ≈30k LOC new dep for a feature that needs
    interpolation, not expression evaluation.

- **Q10 — OpenAPI viewer assets.** Two options:
  - **Hand-rolled tiny viewer** (Part J.2). PINNED — ≈8 KB embedded.
  - `swagger-ui-dist` — rejected, ≈3 MB unpacked binary inflation.

- **Q11 — Postgres driver choice.** Two options:
  - **`jackc/pgx/v5`** (≈9k LOC, Apache-2.0, modern Go, `database/sql`
    compatible via `pgx/stdlib`). PINNED for v0.5.0.
  - `lib/pq` — rejected: in maintenance-only mode upstream; new features
    land in pgx.

- **Q12 — Connection-log `top_source_ips` privacy.** The hourly rollup
  webhook event leaks per-IP session counts. Operators with strict PII
  posture can disable the field via a new setting
  `connection_logs.rollup_include_top_ips` (default true; can be set
  false to omit `top_source_ips` from rollup events and from the
  `connection_log_rollups` storage). The integration plan validates
  both modes.

- **Q13 — Postgres failure recovery.** v0.5.0 ships with no Postgres
  retry/backoff in the migration runner; an unavailable Postgres at
  startup is a fatal error (operator must restart Burrow once the DB is
  reachable). Auto-reconnect / migration retry deferred to v1.0+.
  Honest position for an alpha-flagged feature.
