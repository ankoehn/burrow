# Burrow v0.2.0 Backend API Contract

> Source of truth for the UI and integration agents. Every shape below is copied
> from the implemented handlers in `internal/api`. The code is authoritative; if
> this doc and the code disagree, the code wins — open a fix to this doc.

## Conventions

- **Base path:** `/api/v1` (health probes are the only exception — no prefix).
- **Auth:** session is the `burrow_session` cookie (HttpOnly), set by `POST /api/v1/auth/login`.
- **CSRF:** double-submit. `POST /api/v1/auth/login` also sets `burrow_csrf`
  (JS-readable). Every **non-safe** method (POST/PUT/PATCH/DELETE) on a
  session-protected route MUST send header `X-CSRF-Token` equal to the
  `burrow_csrf` cookie value, or the request gets `403 {"error":"csrf token invalid"}`.
  Safe methods (GET/HEAD/OPTIONS) are exempt.
- **Auth ordering:** unauthenticated → `401 {"error":"unauthorized"}` (checked
  before CSRF and before the admin check). Non-admin on an admin route →
  `403 {"error":"admin required"}`.
- **Error body:** always `{"error":"<message>"}` with the documented status.
- **Content type:** request and response bodies are `application/json`.
- **Auth model note:** authorization keys off the `role` string (`admin`/`user`).
  The `internal/authz` permission map (surfaced read-only at `/roles/{name}`) is
  the extension seam for custom roles in a later release; it is not yet
  user-editable and does not change request authorization in v0.2.0.

## Health (public — no `/api/v1` prefix, no auth, no CSRF)

| Method | Path | Success | Failure |
|---|---|---|---|
| GET | `/healthz` | `200 {"status":"ok"}` | — (always 200 if process is up) |
| GET | `/readyz` | `200 {"status":"ready"}` | `503 {"error":"not ready: database unreachable"}` (also 503 with `not ready: database not configured` if no DB) |

## Auth

### POST `/api/v1/auth/login` — public (rate-limited)
Request: `{ "email": "...", "password": "..." }`
- `200 {"id","email","role"}` + sets `burrow_session` & `burrow_csrf` cookies. Also stamps the user's `last_login` (best-effort).
- `400 {"error":"invalid request body"}` — missing/garbled body or empty email.
- `401 {"error":"invalid credentials"}` — bad email/password.
- `403 {"error":"account suspended"}` — **new in v0.2.0**: user's `status` is `suspended`.
- `429 {"error":"too many login attempts"}` — per-IP or global rate limit.
- `500 {"error":"login failed"}`.

### POST `/api/v1/auth/logout` — session + CSRF
`204`, clears both cookies. (Unchanged.)

### POST `/api/v1/auth/change-password` — session + CSRF
Request: `{ "current_password": "...", "new_password": "..." }`
- `204` success; `400` missing fields / new password < 8 chars; `401` wrong current password; `500` failure. (Unchanged.)

### GET `/api/v1/me` — session
`200 {"id","email","role"}`. (Unchanged.)

## Client tokens (session + CSRF; owner-scoped) — unchanged, documented for completeness

| Method | Path | Request | Success | Errors |
|---|---|---|---|---|
| GET | `/api/v1/tokens` | — | `200 [{"id","name","last_used","created_at"}]` | `500` |
| POST | `/api/v1/tokens` | `{"name":"..."}` | `201 {"name","token"}` (plaintext shown once) | `400` name required; `500` |
| DELETE | `/api/v1/tokens/{id}` | — | `204` | `404` token not found; `500` |

## Tunnels (session + CSRF; owner-scoped)

### GET `/api/v1/tunnels` — session
`200 [{"id","name","type","remote_port","local_addr","bytes_in","bytes_out","connected"}]` (live registry view; unchanged).

### PUT `/api/v1/tunnels/{id}/access-mode` — session + CSRF — **new**
Request: `{ "access_mode": "open" | "api_key" | "burrow_login" }`
Scoped to the caller (own tunnels only).
- `204` success.
- `400 {"error":"access_mode is required"}` — empty/missing field.
- `400 {"error":"access_mode must be 'open', 'api_key', or 'burrow_login'"}` — invalid enum value.
- `404 {"error":"tunnel not found"}` — unknown tunnel or not owned by caller.
- `500` on other failure.

**Runtime note:** only `open` has runtime effect in v0.2.0. `api_key` and
`burrow_login` are accepted and persisted but **inert** (no enforcement) until
HTTP tunnels land in v0.3.0.

## Sessions (session + CSRF; caller-scoped) — **new**

### GET `/api/v1/sessions`
`200 [{"id","ip","user_agent","created_at","expires_at","current"}]` — newest first; `current:true` marks the session making the request.

### DELETE `/api/v1/sessions/{id}`
Revoke one of the caller's sessions. `204`; `404 {"error":"session not found"}` (missing or owned by another user); `500`.

### POST `/api/v1/sessions/revoke-all`
Sign out of every **other** session (keeps the current one). `200 {"revoked": <int>}`; `500`.

## Users (admin only: session + CSRF + admin)

### GET `/api/v1/users?q=&limit=&offset=` — **shape changed in v0.2.0**
- `q` — case-insensitive email substring (default: all).
- `limit` — default 50, capped at 200 (invalid/empty → 50).
- `offset` — default 0 (invalid/empty → 0).

Response `200`:
```json
{
  "users": [
    { "id": "...", "email": "...", "role": "admin|user",
      "status": "active|suspended", "last_login": "RFC3339 or null",
      "created_at": "RFC3339" }
  ],
  "total": 0
}
```
`total` is the full filtered count (ignoring limit/offset). Never includes `password_hash`. Errors: `500`.

### POST `/api/v1/users` — unchanged
Request: `{ "email","password","role" }`
- `201` → a `userAdminResp` object (now also carries `status` (`"active"`) and `last_login` (`null`) for a fresh user).
- `400` email/password/role required; `400` password < 8 chars; `400` role must be 'admin' or 'user'; `409 {"error":"email already in use"}`; `500`.

### PATCH `/api/v1/users/{id}` — **new**
Request (at least one field): `{ "role"?: "admin|user", "status"?: "active|suspended" }`
- `204` success.
- `400 {"error":"role and/or status required"}` — empty body / neither field.
- `400 {"error":"status must be 'active' or 'suspended'"}` / `400 {"error":"role must be 'admin' or 'user'"}` — invalid value.
- `400 {"error":"cannot change your own status"}` — self-status-change lockout guard (an admin may not suspend/activate themselves; role self-change is allowed).
- `404 {"error":"user not found"}`; `500`.

**Side effect:** setting `status:"suspended"` immediately revokes all of that
user's sessions (suspension takes effect without a per-request user lookup).

### DELETE `/api/v1/users/{id}` — unchanged
`204`; `400 {"error":"cannot delete yourself"}`; `404 {"error":"user not found"}`; `500`.

## Roles (admin only) — **new, read-only**

### GET `/api/v1/roles`
`200 [{"name","description","created_at"}]` (built-in `admin`, `user`).

### GET `/api/v1/roles/{name}`
`200 {"name","description","created_at","permissions":[...]}` — `permissions`
is the code-defined `internal/authz` set for that role (always a JSON array,
possibly empty). `404 {"error":"role not found"}` for unknown names; `500`.

**Permission vocabulary** (`internal/authz`; `:own` = act on resources you own,
`:any` = act on anyone's; global capabilities have no scope suffix):

`tunnels:read:own`, `tunnels:read:any`, `tunnels:manage:own`,
`tunnels:manage:any`, `tokens:manage:own`, `tokens:manage:any`,
`services:configure:own`, `services:configure:any`, `sessions:manage:own`,
`sessions:manage:any`, `users:read`, `users:manage`, `roles:read`,
`settings:manage`.

Built-in mappings:
- **admin** → every permission above (full `:any` access).
- **user** → `tunnels:read:own`, `tunnels:manage:own`, `tokens:manage:own`,
  `services:configure:own`, `sessions:manage:own`.

## Settings (admin only) — **new**

### GET `/api/v1/settings`
`200` — a flat map of **whitelisted non-secret** keys only:
`smtp.host`, `smtp.port`, `smtp.username`, `smtp.from`, `smtp.tls`.
Only keys that are set appear. **`smtp.password` is never returned.**

### PUT `/api/v1/settings`
Request: a flat `{ "key": "value" }` map. Only whitelisted keys are persisted;
every other key (including `smtp.password`) is silently dropped. `smtp.tls`, if
present, must be `none` | `starttls` | `implicit`.
- `204` success.
- `400 {"error":"invalid request body"}` — unparseable body.
- `400 {"error":"smtp.tls must be none, starttls, or implicit"}`.
- `500`.

### POST `/api/v1/settings/test-email`
Request: `{ "to": "ops@example.com" }` — sends a real SMTP test message using
the stored non-secret config plus the secret password from the environment.
- `204` success.
- `400 {"error":"to is required"}`.
- `409 {"error":"smtp is not configured — set host/port and BURROW_SMTP_PASSWORD"}` — no `smtp.host` configured.
- `502 {"error":"test email failed: <detail>"}` — dial/TLS/auth/protocol failure.

**Secret handling (hard rule):** the SMTP password is sourced exclusively from
`BURROW_SMTP_PASSWORD` or `BURROW_SMTP_PASSWORD_FILE` (the `_FILE` form wins,
Docker-secret convention) and is **never** read or written through the API or
stored in the `settings` table. `smtp.tls` defaults to `starttls` when unset.

## Clients (admin only) — **new**

A *client* = one live authenticated control session (one `burrow` process); a
*service* = one tunnel that client serves.

### GET `/api/v1/clients`
`200 [ClientView]`:
```json
{
  "session_id": "...", "user_id": "...", "token_name": "...",
  "remote_addr": "ip:port", "os": "linux", "arch": "amd64",
  "client_version": "...", "service_count": 0,
  "total_bytes_in": 0, "total_bytes_out": 0
}
```
`total_bytes_*` is the **persisted** aggregate across the client's services
(survives reconnects). `token_name`/`os`/`arch`/`client_version` are
best-effort from the handshake (empty for clients predating the metadata).

### GET `/api/v1/clients/{sessionID}`
`200 ClientDetail` = a `ClientView` (same fields, inlined) plus:
```json
"services": [
  { "id":"...", "name":"...", "type":"tcp", "remote_port":9000,
    "local_addr":"127.0.0.1:3000", "access_mode":"open",
    "bytes_in":0, "bytes_out":0,
    "total_bytes_in":0, "total_bytes_out":0 }
]
```
`bytes_in`/`bytes_out` are **live** (since the current connect);
`total_bytes_*` are **persisted** (survive reconnects). `services` is always a
JSON array (possibly empty). `404 {"error":"client not found"}` for an unknown
or disconnected session id.

## Protocol note

`proto.AuthRequest` has a reserved optional `hostname` field
(`"hostname,omitempty"`, comment `// hostname (optional, since v0.3 extension)`).
It is **unused in v0.2.0** — integration agents and clients MUST NOT depend on
it carrying meaning yet.

## Deferred (NOT in v0.2.0 — do not build UI/integration against these)

- Provisioning keys / pending-client approval — v0.3.0.
- `api_key` / `burrow_login` access-mode **runtime enforcement** — v0.3.0
  (the field is settable now but inert).
- HTTP tunnels / vhost routing / Let's Encrypt — v0.3.0.
